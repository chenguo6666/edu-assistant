## 一、Sprint Goal 达成情况

### 原始目标

> 将系统从"通用教育助手"升级为"保研专属 AI 导师"，新增学生档案管理、保研知识库、竞争力评估、院校推荐与申请规划能力。

### 成功标准对比

| # | 成功标准 | Sprint 3 Backlog 定义 | 实际交付 | 状态 |
|---|----------|----------------------|----------|------|
| 1 | 用户可以创建和编辑保研档案 | GPA、排名、英语、科研、获奖、目标院校 | GPA、年级、专业、学科（无排名/英语/科研/获奖） | ⚠️ 部分 |
| 2 | 用户可以提交"评估竞争力"请求 | 结构化评估报告（strengths/weaknesses/suggestions） | 条件匹配检索（非结构化报告） | ⚠️ 调整 |
| 3 | 用户可以提交"推荐院校"请求 | 分组推荐（冲刺/稳妥/保底，各 2 所） | 院校信息查询（无分组） | ⚠️ 调整 |
| 4 | 用户可以提交"申请规划"请求 | 时间线 + 材料清单 | 预推免时间线查询 | ✅ 基本达成 |
| 5 | 保研知识库内置院校招生信息 | 1000+ 条记录 | 12 所高校，20 条向量记录 | ⚠️ 数据量不足 |

**结论**：Sprint 3 核心功能已实现，但部分功能的实现方式与原始 backlog 有差异。

## 五、与 Sprint 3 Backlog 的差异分析

### 需要说明的调整

| Backlog 需求 | 实际实现 | 调整原因 |
|-------------|----------|----------|
| 学生档案 9 个字段 | 4 个字段（GPA/年级/专业/学科） | 简化 MVP，聚焦核心信息 |
| 竞争力评估报告 | 条件匹配检索 | 结构化报告需要更复杂的 LLM Prompt |
| 冲刺/稳妥/保底分组 | 统一检索返回 | 分组逻辑需要额外的评分算法 |
| 1000+ 知识库记录 | 12 所高校 20 条记录 | 数据采集工作量大，聚焦 985/211 |
| 独立保研档案表 | 复用 users 表 | 简化数据模型 |

### 超额交付

| 功能 | 说明 |
|------|------|
| WebSocket 流式 | 原始 Backlog 未要求，实际实现了实时推送 |
| 联网搜索 | 原始 Backlog 未要求，实际集成了 SerpAPI |
| 答题交互 | 原始 Backlog 未要求，实际实现了选择题解析与作答 |
| 导出功能 | 原始 Backlog 未要求，实际支持 Markdown + PDF |
| 用户画像注入 | 原始 Backlog 未要求，实际实现了个性化 Prompt |

## 七、风险与局限

| 风险 | 影响 | 现状 |
|------|------|------|
| 知识库数据量不足 | 保研查询覆盖面有限 | 12 所高校，建议后续扩充 |
| LLM 响应速度 | 取决于硅基流动 API | 397B 模型较慢但稳定 |
| SerpAPI 额度限制 | 联网搜索受限 | 免费 100 次/月 |
| 条件匹配非结构化 | 无法生成评估报告 | 简化为 RAG 检索 |
